Welcome to the MCP Hands-on Lab. Today we'll learn how to give AI assistants access to real tools — from running commands on remote machines to querying databases.

The Problem

AI models are isolated from your tools and data.

They can't read your files, run your tests, check your servers, or access your APIs.

Without MCP, every AI integration is a custom one-off. Each tool needs its own API adapter, authentication flow, and data format. This doesn't scale.

The Solution: MCP

A universal, open protocol connecting AI assistants to external tools.

MCP standardizes the connection. Build a tool once, use it with Claude, VS Code, Cursor, or any MCP-compatible client. Think of it like USB for AI tools.

Architecture

  • Host: Claude Code, VS Code, Cursor
  • Client: Built into the host, manages server connections
  • Server: Your code — exposes tools, resources, prompts
The host application contains an MCP client that can connect to multiple servers. Communication uses JSON-RPC 2.0 over either stdio or HTTP.

The Three Primitives

PrimitiveWhatWho ControlsExample
Tools Functions the AI can call AI decides when to use run_remote_command(host, cmd)
Resources Read-only data User/app controls access file://config.yaml
Prompts Instruction templates User selects security_review checklist
Tools are the most common primitive. The AI reads the tool description and decides when to call it. Resources are for data. Prompts are pre-built instructions.

How MCP Works

Communication Flow

sequenceDiagram participant C as Client participant S as Server C->>S: initialize (protocol version, capabilities) S->>C: initialize response (server capabilities) C->>S: tools/list S->>C: [tool definitions with JSON schemas] Note over C: User asks a question... C->>S: tools/call {name: "port_checker", args: {ports: "8080"}} S->>C: {content: [{type: "text", text: ":8080 IN USE (pid 1234)"}]}
The client discovers tools on startup, then calls them as needed. All communication uses JSON-RPC 2.0 messages. The protocol is stateful — there's an initialization handshake.

Transport: STDIO vs HTTP

STDIO (Local)

graph TB C[Client] -->|stdin| S[Server Process] S -->|stdout| C style C fill:#2ecc71,stroke:#27ae60,color:#fff style S fill:#3498db,stroke:#2980b9,color:#fff
  • Same machine only
  • One client per server
  • Zero network config
  • Best for local dev tools

HTTP (Remote)

graph TB C1[Client 1] -->|HTTPS| S[Server :9000] C2[Client 2] -->|HTTPS| S C3[Client 3] -->|HTTPS| S style C1 fill:#2ecc71,stroke:#27ae60,color:#fff style C2 fill:#2ecc71,stroke:#27ae60,color:#fff style C3 fill:#2ecc71,stroke:#27ae60,color:#fff style S fill:#e74c3c,stroke:#c0392b,color:#fff
  • Accessible from anywhere
  • Multiple clients
  • Needs auth + TLS
  • Best for team/cloud tools
STDIO is simpler — the host spawns the server as a child process and communicates via pipes. HTTP runs on a port and can serve multiple clients over the network.

Under the Hood: JSON-RPC

All MCP messages are JSON-RPC 2.0:

// Client → Server: Call a tool
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tools/call",
  "params": {
    "name": "port_checker",
    "arguments": { "ports": "3000,8080" }
  }
}

// Server → Client: Tool result
{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "content": [{
      "type": "text",
      "text": "Port Status:\n  :3000  FREE\n  :8080  IN USE"
    }]
  }
}
You don't need to write JSON-RPC manually — the SDK handles it. But understanding the wire format helps when debugging.

Building MCP Servers with Python

FastMCP: 10 Lines to Your First Server

from fastmcp import FastMCP

mcp = FastMCP("my-tools")

@mcp.tool()
def greet(name: str) -> str:
    """Say hello to someone."""
    return f"Hello, {name}!"

if __name__ == "__main__":
    mcp.run(transport="stdio")
That's it. Type hints → JSON schema. Docstring → tool description. One decorator → registered.
FastMCP is the official high-level Python SDK. It uses decorators to register tools and automatically generates JSON schemas from your type hints.

Tools: @mcp.tool()

@mcp.tool()
def run_remote_command(host: str, command: str, user: str = "") -> str:
    """Run a shell command on a remote machine via SSH.

    Args:
        host: Hostname or IP address.
        command: The shell command to execute.
        user: SSH username (optional).
    """
    ssh_target = f"{user}@{host}" if user else host
    result = subprocess.run(
        ["ssh", ssh_target, command],
        capture_output=True, text=True, timeout=30,
    )
    return result.stdout or result.stderr

The AI reads the docstring to decide when and how to call this tool.

Good docstrings are critical. The AI uses them to understand what the tool does, what parameters mean, and when it's appropriate to call it.

Prompts: @mcp.prompt()

@mcp.prompt()
def security_review() -> str:
    """Security-focused review checklist."""
    return """You are performing a security review. Focus on:

    ## Authentication & Authorization
    - Are auth checks present on all protected endpoints?
    - Is session management secure?

    ## Input Validation
    - Is all external input validated?
    - Is output encoding applied?

    ## OWASP Top 10
    - Injection, XSS, CSRF, ...

    Flag issues by severity: CRITICAL / HIGH / MEDIUM / LOW."""

Prompts give the AI rich context for specific tasks.

Prompts are like instruction templates. They're not called automatically — the user asks the AI to use them. Great for standardizing review processes.

Resources: @mcp.resource()

@mcp.resource("config://app")
def get_app_config() -> str:
    """Returns the application configuration."""
    return Path("config.yaml").read_text()

@mcp.resource("logs://{service}")
def get_service_logs(service: str) -> str:
    """Returns recent logs for a service."""
    return Path(f"/var/log/{service}.log").read_text()[-5000:]

Resources expose read-only data with URI templates.

Resources are like GET endpoints. The AI or user can read them but can't modify them. URI templates with curly braces create dynamic resources.

Testing: FastMCP Inspector

# Launch the web inspector
fastmcp dev my_server.py

Opens a web UI where you can:

  • See all registered tools, resources, prompts
  • Call tools with custom arguments
  • View raw JSON-RPC messages
  • Debug before connecting to any AI client
Always test with the inspector first. It shows you exactly what the AI client will see — tool schemas, descriptions, and responses.

Connecting to Claude Code

Create .mcp.json in your project root:

{
  "mcpServers": {
    "dev-tools": {
      "command": "python",
      "args": ["path/to/server.py"]
    }
  }
}
# Start Claude Code from the project directory
cd my-project
claude
# → Tools are auto-discovered!
Claude Code reads .mcp.json on startup and launches each server as a child process. The tools appear automatically in the conversation.

Connecting to VS Code

Create .vscode/mcp.json in your workspace:

{
  "mcpServers": {
    "dev-tools": {
      "command": "python",
      "args": ["path/to/server.py"]
    }
  }
}
  • Restart VS Code after editing
  • Open Copilot Chat → look for the tools icon
  • Requires VS Code 1.99+
Same format as Claude Code. VS Code's Copilot Chat will show MCP tools in the tools panel. You can also use variable substitution like ${workspaceFolder}.

External MCP Servers

Connect to community-built servers via npx:

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "."]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_..."
      }
    }
  }
}
You can mix and match — your custom servers alongside community servers. Each server runs as an independent process.

Security Considerations

MCP servers run with your permissions. They can access your files, network, and credentials.

  • Least privilege: Only expose needed directories/APIs
  • No hardcoded secrets: Use environment variables
  • Review source code: Know what a server does before connecting
  • Pin versions: Use specific package versions
  • Monitor: Check what tools are being called
This is important to emphasize. MCP servers run as local processes with the user's permissions. A malicious server could access anything the user can access.

Authentication in Production

Options

  • API Key (header)
  • Azure Entra ID (OAuth)
  • Bearer tokens
  • mTLS (certificates)

Config

{
  "mcpServers": {
    "remote": {
      "url": "https://...",
      "headers": {
        "X-API-Key": "${env:KEY}"
      }
    }
  }
}
Never expose an MCP server to the internet without authentication.
